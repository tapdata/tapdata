# DuckDB 状态快照存储到 MongoDB GridFS 方案设计草案

## 1. 文档目的

本文档用于整理 Tapdata 在多引擎环境下，为包含 DuckDB 节点的任务设计“DuckDB 状态快照存储到 MongoDB GridFS”的高可用恢复方案。

本文档当前定位为第一版设计草案，目标不是一次性给出全部实现细节，而是先把以下内容结构化沉淀下来，作为后续逐章讨论和收敛的基础：

- 明确该 HA 方案要解决的业务问题与故障转移场景
- 明确该方案的设计边界、非目标与前置假设
- 明确方案的总体流程、核心组件与存储模型
- 明确后续必须优先讨论的关键问题，避免文档直接进入实现细枝末节

## 2. 背景与问题定义

Tapdata 当前支持在任务中运行 DuckDB 节点。DuckDB 节点在运行过程中会在引擎本地维护数据库文件目录，用于保存缓存表、宽表及相关运行状态。

在多引擎部署场景下，任务调度可能发生故障转移。典型场景是：

- 任务最初运行在引擎节点 A
- 任务运行期间，DuckDB 节点持续在节点 A 的本地目录中写入数据文件
- 当节点 A 故障、任务被调度漂移到节点 B 时，节点 B 无法直接访问节点 A 的本地 DuckDB 文件
- 如果没有额外恢复机制，任务虽然可以在节点 B 上被重新拉起，但 DuckDB 本地状态无法继承，进而可能导致任务无法正常恢复

当前前提是没有现成共享文件存储可用，因此本方案不再依赖共享目录，而是把 DuckDB 可恢复状态打包为任务级快照，存入 MongoDB GridFS，由接管节点在恢复时下载并解压到本地目录。

## 3. 设计目标

本方案的设计目标如下：

- 支持 Tapdata 多引擎环境下，包含 DuckDB 节点的任务发生故障转移后，在新引擎节点上恢复 DuckDB 本地状态
- 支持把 DuckDB 本地目录在一致性点生成任务级快照，并按任务维度写入 MongoDB GridFS
- 支持在任务被 Tapdata 现有恢复机制拉起时，从 MongoDB GridFS 下载最新可恢复快照并还原到当前引擎节点的本地目录
- 支持通过普通 Mongo 集合维护快照元数据、状态、校验信息和恢复记录
- 为后续实现一致性校验、异常恢复、监控告警、验收测试提供清晰的设计边界

## 4. 非目标与边界

为避免方案边界模糊，本文档先明确当前不覆盖的范围。

### 4.1 非目标

- 不支持两个引擎节点同时对同一任务的同一份 DuckDB 数据目录进行并发写入
- 不设计为 DuckDB 集群，不提供数据库级多主复制能力
- 不把 MongoDB GridFS 设计成“实时共享目录”
- 不在第一阶段实现文件级增量同步、块级增量同步或日志级复制
- 不承诺当前方案天然实现零数据丢失；是否支持 `RPO=0` 需要基于一致性机制进一步评估

### 4.2 适用边界

- 仅面向“单任务单活”的故障转移场景
- 仅面向“任务漂移到另一台引擎节点后，DuckDB 本地状态可恢复”的高可用诉求
- 仅在用户已指定 DuckDB 本地目录的前提下讨论持久化快照
- 仅在 MongoDB 可访问且 GridFS 可用的前提下讨论快照上传与恢复

## 5. 核心思路概述

本方案的核心思想是：任务运行期间继续使用当前引擎本地 DuckDB 目录作为实际工作目录，不直接把 MongoDB GridFS 当成共享文件系统使用，而是在 DuckDB 完成一次事件处理并成功 `accept` 下游事件后，由快照守护线程感知目录变更并生成任务级快照，按任务 ID 和快照版本写入 GridFS，并在普通 Mongo 集合中记录快照元数据。

当任务发生故障转移并漂移到新的引擎节点后，由 Tapdata 现有任务恢复机制负责重新拉起任务实例；DuckDB HA 方案只负责 DuckDB 本地状态恢复。接管节点在 DuckDB 节点初始化前，从 MongoDB 中查找最新一个可恢复快照，下载后解压到本地 DuckDB 目录，再把恢复后的本地目录交给任务实例继续启动和使用。

换言之，这不是“共享目录实时运行方案”，而是“本地运行 + 事件处理完成后触发快照 + 基础包/增量包写入 GridFS + 故障后快照恢复”的方案。

## 6. 主流程描述

### 6.1 正常运行阶段

任务在当前引擎节点运行时：

1. 根据任务配置创建或定位当前任务对应的本地 DuckDB 目录
2. DuckDB 节点继续以该本地目录作为实际工作目录
3. DuckDB 完成一次增量事件处理，并且相关事件已经成功 `accept` 到下游后，触发快照守护线程进入待执行状态
4. 快照守护线程定期扫描 DuckDB 本地目录，检测自上一个已提交快照以来是否发生目录变更
5. 检测到目录变更后，守护线程按快照策略生成基础包或增量包
6. 快照归档包按任务 ID 和快照 ID 上传到 MongoDB GridFS
7. 同步写入快照元数据，包括状态、时间、校验值、文件大小、来源引擎、基础包引用或父增量包引用等信息
8. 仅当快照上传和元数据提交全部成功时，该快照才标记为可恢复

### 6.2 故障转移恢复阶段

任务漂移到新的引擎节点后，由 Tapdata 现有任务恢复机制负责重新拉起任务实例；DuckDB HA 方案只负责 DuckDB 本地状态恢复，恢复阶段执行以下动作：

1. 重置当前引擎节点上该任务对应的本地 DuckDB 目录
2. 根据任务 ID 查询 Mongo 中该任务最新一个可恢复快照链
3. 先确定恢复所需的基础包，再按顺序定位需要回放的增量包集合
4. 从 MongoDB GridFS 下载基础包和对应增量包到本地临时目录
5. 校验快照状态、包顺序、大小和校验值
6. 先解压基础包，再按顺序回放增量包到当前引擎节点的本地 DuckDB 目录
7. 完成恢复后执行必要的本地目录校验
8. 校验通过后，把 DuckDB 本地目录交给任务实例继续启动和使用

## 7. 术语定义

为了便于后续讨论，本文档统一采用以下术语：

- 本地 DuckDB 目录：任务当前运行节点上实际被 DuckDB 使用的工作目录
- 快照守护线程：负责在增量事件处理完成后感知快照触发条件、定期扫描目录变更并执行快照生成和上传的后台线程
- 快照归档包：把 DuckDB 本地目录状态打包压缩后形成的单个归档文件
- 快照仓库：MongoDB GridFS 中存储 DuckDB 快照归档包的逻辑区域
- 快照元数据：普通 Mongo 集合中记录的快照状态、校验值、来源节点等描述信息
- 可恢复快照：已经完整上传、元数据已提交且通过基本校验的快照版本
- 基础包：某一时点完整覆盖 DuckDB 本地目录状态的全量快照包
- 增量包：相对于某个基础包或前一增量包，仅记录目录差异的快照包
- 接管节点：在故障转移后，由 Tapdata 现有恢复机制拉起该任务实例的新引擎节点
- 恢复窗口：任务最新本地状态与最近一个可恢复快照之间的时间差

## 8. 总体架构拆分

为了让后续实现职责清晰，建议把方案拆成以下几个组件。

### 8.1 恢复衔接层

职责：

- 与 Tapdata 现有任务恢复机制对接，在 DuckDB 节点初始化前插入状态恢复动作
- 明确 DuckDB HA 只负责目录恢复，不负责任务级调度、漂移判定和任务生命周期管理
- 在恢复完成后，把可用的本地 DuckDB 目录交回现有任务启动链路

### 8.2 本地目录管理器

职责：

- 管理本地 DuckDB 目录的创建、清理、重置与校验
- 统一封装任务级目录路径规则，避免目录污染
- 在恢复阶段负责本地目录重建前的准备动作

### 8.3 快照生成器

职责：

- 在满足一致性条件时生成 DuckDB 状态快照
- 根据目录变更情况决定生成基础包还是增量包
- 执行目录打包、压缩、差异计算、校验和计算
- 生成快照 ID、归档格式、文件统计、父包引用等信息

### 8.4 GridFS 上传器

职责：

- 把快照归档包写入 MongoDB GridFS
- 维护上传状态、失败重试和临时资源清理
- 返回 GridFS 文件 ID 和写入结果

### 8.5 快照元数据管理器

职责：

- 记录任务级快照元数据，例如快照状态、时间、大小、校验值、来源引擎、GridFS 文件 ID 等
- 提供“查询最新可恢复快照”“标记失败快照”“清理过期快照”的能力
- 为故障排查、监控告警和后续恢复判断提供依据

### 8.6 恢复执行器

职责：

- 在接管节点执行快照下载和本地恢复
- 从 MongoDB GridFS 下载快照归档包到本地临时目录
- 完成解压、校验，并向恢复衔接层返回恢复结果

## 9. 存储设计

### 9.1 存储分层原则

本方案采用“两层存储”：

- `GridFS` 负责存放快照归档包二进制内容
- 普通 `Mongo collection` 负责存放快照元数据和恢复控制信息

这个边界必须明确：

- GridFS 存的是“恢复内容”
- 普通集合存的是“恢复描述信息”

### 9.2 GridFS 存储内容

GridFS 中存放每次生成的 DuckDB 快照归档包。建议逻辑上按 `taskId` 管理，但物理上依然使用 GridFS 的文件记录机制，不要求在数据库中真实创建目录结构。

GridFS 中的快照归档包分为两类：

- 基础包：保存某个时点 DuckDB 本地目录的完整状态
- 增量包：保存相对于基础包或前一增量包的目录差异

建议快照文件命名包含以下信息：

- `taskId`
- `snapshotId`
- `snapshotType`
- `engineId`
- `timestamp`
- `archiveFormat`

建议文件名示例：

- `duckdb-ha_<taskId>_<snapshotId>_base_<timestamp>.tar.zst`
- `duckdb-ha_<taskId>_<snapshotId>_delta_<timestamp>.tar.zst`

### 9.3 快照元数据集合

建议新增独立集合，例如 `DuckDbHaSnapshotMeta`，用于记录每个快照版本的元数据。

建议字段至少包括：

- `taskId`
- `snapshotId`
- `snapshotType`
- `engineId`
- `status`
- `createdAt`
- `completedAt`
- `archiveFormat`
- `archiveSize`
- `checksum`
- `gridFsFileId`
- `isRecoverable`
- `failureReason`
- `sourceDbPath`
- `fileCount`
- `baseSnapshotId`
- `parentSnapshotId`
- `sequenceNo`
- `restoreCount`
- `lastRestoreAt`

### 9.4 快照状态机

建议快照生命周期至少包含以下状态：

- `PREPARING`：准备生成快照
- `PACKING`：正在打包压缩
- `UPLOADING`：正在上传到 GridFS
- `COMMITTED`：上传和元数据提交成功，可用于恢复
- `FAILED`：快照生成或上传失败，不可用于恢复
- `EXPIRED`：快照过期，保留记录但不再用于恢复

恢复流程只能选择 `COMMITTED` 且 `isRecoverable=true` 的快照。若选择的是增量包，则必须能沿 `baseSnapshotId` 和 `parentSnapshotId` 回溯出完整、连续、未缺失的恢复链。

## 11. 当前方案的关键设计前提

为便于后续讨论，当前先把该思路隐含的前提条件显式写出。

### 11.1 单任务单活前提

该方案默认 Tapdata 现有任务调度与恢复机制已经保证同一任务在任一时刻仅允许一个引擎节点实际运行。DuckDB HA 方案是在这个前提下补充 DuckDB 本地状态恢复能力。如果这一前提不成立，则同一任务可能产生多个不一致快照版本，方案成立基础会被破坏。

### 11.2 MongoDB GridFS 可访问前提

该方案默认所有可能接管任务的引擎节点都能访问同一个 MongoDB 集群，并且该集群具备：

- 足够的可用性
- 可接受的网络吞吐
- 满足任务快照规模的存储容量
- 能支撑 GridFS 大文件上传下载的基本性能

### 11.3 事件驱动 + 定期扫描前提

当前思路本质上是“增量事件处理完成后触发快照候选 + 守护线程定期扫描目录变更 + 按需生成快照”，而不是每次 DuckDB 写入都同步写入 MongoDB。因此系统天然存在恢复窗口，后续需要基于扫描周期、基础包/增量包策略和一致性模型定义对应的 `RPO`。

### 11.4 恢复前重置本地目录前提

接管节点在恢复前，需要先重置本地该任务对应的 DuckDB 目录，避免：

- 本地残留旧文件污染恢复结果
- 目录中存在与目标快照不一致的历史文件
- 恢复后 DuckDB 打开混合状态目录

## 12. 当前识别出的关键风险

虽然整体思路清晰，但该方案能否真正落地，核心取决于下面几个问题能否设计清楚。

### 12.1 快照一致性风险

DuckDB 在运行中可能同时维护数据库主文件、预写日志文件、临时文件或其他运行时相关文件。若快照在错误时机打包，可能得到逻辑上不自洽的文件组合。此时归档包虽然上传成功，但未必是一个可恢复的数据库状态。

### 12.2 快照窗口导致的数据回退风险

由于方案采用“事件驱动候选 + 定期扫描落盘”的快照机制而不是实时同步，故障发生时只能恢复到最近一个成功快照链。最近一次 `accept` 之后但尚未被守护线程扫描、打包并成功提交到 GridFS 的本地状态可能丢失。

### 12.3 Mongo 容量与写入压力风险

DuckDB 目录可能较大。如果快照频率高、任务多或基础包体积大，会显著增加 Mongo 存储容量、GridFS 分块写入量和备份压力。即使引入增量包，也需要控制增量链长度，否则恢复成本会上升。

### 12.4 压缩与恢复时延风险

打包压缩和下载解压都是重操作。当本地目录较大或网络较慢时，会直接影响快照生成耗时和故障恢复时长。

### 12.5 快照清理与版本堆积风险

如果没有快照保留和淘汰策略，GridFS 中会快速堆积大量历史基础包和增量包，影响容量和后续运维。删除时还需要保证不会误删仍被增量链依赖的基础包。

## 13. 建议优先讨论顺序

为了让后续讨论更高效，建议不要按文档章节顺序平均展开，而是优先讨论最影响方案可行性的核心问题。

### 13.1 第一优先级：一致性模型与可恢复点

优先回答以下问题：

- DuckDB 在什么状态下允许生成快照
- 快照前是否必须执行 checkpoint、flush 或短暂冻结写入
- 哪些文件必须被一起打包
- 什么条件下才能把快照标记为 `COMMITTED`

### 13.2 第二优先级：快照生成与上传机制

优先回答以下问题：

- 快照采用“增量事件处理完成后触发候选 + 守护线程定期扫描目录”的组合方式后，如何避免频繁空跑
- 基础包和增量包分别使用什么归档格式
- 打包和上传失败后如何重试
- 是否允许同一任务存在并行快照

### 13.3 第三优先级：恢复流程与失败处理

优先回答以下问题：

- 如何选择最新可恢复快照
- 下载失败、解压失败、校验失败如何处理
- 恢复后如何验证本地 DuckDB 目录可用

### 13.4 第四优先级：容量治理与清理策略

优先回答以下问题：

- 每个任务保留多少个历史快照
- 是否按时间、数量或空间占用做淘汰
- 过期快照如何安全删除

### 13.5 第五优先级：配置、监控、测试与验收

优先回答以下问题：

- 面向产品和运维暴露哪些配置
- 如何衡量快照质量与恢复质量
- 如何用测试证明该方案可用

## 14. 需要后续明确的开放问题

以下问题需要在下一轮讨论中逐项收敛。

### 14.1 快照时机

当前已明确：

- 快照触发前提是发生增量，并且 DuckDB 已完成事件处理且成功 `accept` 下游事件
- `accept` 成功后不立即原地打包，而是只唤醒快照守护线程进入待执行状态
- 守护线程通过定期扫描 DuckDB 本地目录变更来决定是否真正生成快照

后续还需要补充：

- 扫描周期具体取值
- 高频 `accept` 场景下如何合并多次触发
- 全量结束后是否额外强制生成一份基础包

### 14.2 快照格式

当前已明确采用“基础包 + 增量包”模式，目标是支持后续增量刷新能力。

后续还需要补充：

- 基础包和增量包是否统一使用 `tar.zst`
- 增量包的差异计算规则
- 何时强制重新生成新的基础包以截断过长增量链

### 14.3 快照校验强度

恢复流程中，是要求“下载后校验和通过才能解压、解压后目录校验通过才能启动任务”，还是允许更宽松的尽力恢复模式。这会直接影响恢复时间与恢复成功率定义。

### 14.4 Mongo 负载上限

需要根据任务数量、DuckDB 目录平均大小、快照频率估算：

- 每日新增快照容量
- GridFS 分块数量
- 上传和下载峰值流量
- 对现有 Mongo 集群的额外压力

### 14.5 适用范围扩展

当前方案是只支持“跨引擎漂移恢复”，还是还要支持：

- 引擎进程重启后的本地恢复
- 同一台机器上的任务重拉起恢复
- 后续更广义的任务热迁移能力

## 16. 当前结论

“本地运行 + 增量事件处理完成并 `accept` 后触发快照守护线程 + 定期扫描目录变更 + 基础包/增量包写入 MongoDB GridFS + 故障后按快照链恢复本地目录”的思路，可以作为当前缺少共享存储前提下的 DuckDB HA 可行候选方案。

但它的真正落地前提是先解决以下两个根问题：

- 生成快照时，如何定义并保证 DuckDB 快照的可恢复一致性
- 恢复时，如何在可接受的 `RPO` 和 `RTO` 范围内完成下载、解压和校验

如果这两个问题无法被严格定义和验证，则即使 GridFS 上传下载链路实现出来，也无法真正构成可靠的高可用方案。
